# 캐릭터 관리 앱 — 개발 지침서

## 문서 목적

이 문서는 캐릭터 관리 앱의 개발 방향과 의사결정 기준을 명문화한 지침서다. 기능 설계, 구현, 리팩토링 등 모든 개발 단계에서 일관성을 유지하기 위해 작성되었으며, 모든 기능 구현 시 이 문서의 원칙을 우선적으로 참조해야 한다.

## 핵심 철학

이 앱은 **"도구를 제공하는 앱"이 아니라 "도구를 만들 수 있는 도구를 제공하는 앱"**이다. 사용자가 앱의 틀 안에 맞추는 것이 아니라, 앱이 사용자의 필요에 맞춘다. 엑셀에서 계산식을 자유롭게 지정하듯, 각 필드의 속성과 동작을 사용자가 정의할 수 있어야 한다. 하지만 단지 사용자가 설정한 것만을 하는 것이 아니라 거기에 더한 편의성과 기능성은 최대한으로 제공한다.

> "개발자가 예상한 범위 안에서만 쓰는 앱이 아니라, 내가 필요한 대로 구조 자체를 정의하고 그 위에서 실질적인 인사이트를 뽑아낼 수 있는 앱"

---

## 5대 개발 원칙

### 원칙 01. 열린 구조 (Open Architecture)

모든 기능은 확장 가능해야 한다. 고정된 틀, 닫힌 프리셋, 확장 불가능한 시스템은 허용되지 않는다.

- 필드는 사용자가 자유롭게 추가/삭제/편집할 수 있어야 한다.
- 프리셋과 템플릿은 불러오기뿐 아니라 추가/편집/삭제가 모두 가능해야 한다.
- 신규 필드가 자동으로 통계 시스템에 편입되고, 분석 방법도 사용자가 설정할 수 있어야 한다.
- 다른 작품에서 사용한 필드를 불러와 재사용할 수 있는 구조를 갖춰야 한다.

### 원칙 02. 실질적 기능성 (Substantive Functionality)

기능의 '존재'가 아니라 '실질적 쓰임'이 기준이다. 겉핥기식 기능은 없는 것보다 못하다.

- 통계는 '데이터 입력량'이 아니라 '분석적 인사이트'를 제공해야 한다. 성별 비율, 특성 분포, 거주지 분포, 사건 연계율 등 실제로 작가가 캐릭터 군상을 조감하고 편향이나 패턴을 발견할 수 있는 정보를 줘야 한다.
- 사건 기능은 연표에서 사건 존재를 한눈에 파악할 수 있어야 한다.
- 모든 필드(머리카락 색, 거주지 등 세부 데이터까지)가 통계에서 분석 가능해야 한다.
- 방편식 패치(임시 땜빵)는 금지한다. 모든 기능은 완결된 형태로 구현한다.

### 원칙 03. 기능 우선, 구조 후순 (Function Over Simplicity)

구조의 간결함과 기능의 풍부함이 충돌할 때, 항상 기능을 선택한다.

- 구조가 복잡해지더라도 기능이 조금이라도 좋으면 그쪽을 채택한다.
- 기능 간소화는 절대 허용되지 않는다.
- '할 수 있는 건 다 한' 수준이 기준선이다.
- 단, UX 복잡도는 이중 경로(러프 입력 / 정밀 조정)로 해결한다.

### 원칙 04. 조작 마찰 최소화 (Minimal Friction)

기능의 깊이는 최대한 깊되, 접근과 조작은 최대한 짧아야 한다.

- 저장을 위해 버튼 여러 단계를 거치는 플로우는 금지한다.
- 러프하게 입력하고 나중에 정밀 조정하는 이중 경로를 제공한다. 예를 들어 사건 배치 시 대략적 위치만 먼저 잡고 세부 조정은 나중에 할 수 있어야 한다.
- 일일이 확인하지 않으면 존재를 알 수 없는 데이터가 있어서는 안 된다.
- 순서 편집 등 반복 작업은 드래그앤드롭 등 직관적 조작을 우선 제공한다.

### 원칙 05. 데이터 유기적 연결 (Organic Data Linkage)

작품–세계관–캐릭터–사건은 고립된 저장소가 아니라 하나의 유기적 데이터베이스처럼 작동해야 한다.

- 필드를 다른 작품에서 불러와 재사용할 수 있어야 한다.
- 세계관/작품 이미지를 캐릭터 이미지에서 자동 연동(랜덤/선택)할 수 있어야 한다.
- 작품 선택 시 관련 캐릭터가 자동 필터링되어야 한다.
- 통계가 작품별로 다른 필드 구조를 인지하고 적응적으로 분석해야 한다.

## 판단 기준: Do / Don't

### 필드 시스템

**Do:**
- 필드의 타입, 파싱 방식, 통계 포함 여부를 각각 설정 가능하게 구현
- 이명/별칭은 콤마로 구분하여 복수 입력 지원
- 다른 작품의 필드를 불러와 재사용할 수 있는 기능 제공
- 이름 필드를 First Name / Last Name으로 분리 가능하게 설계

**Don't:**
- 필드 기능(콤마 파싱 등)을 특정 필드에만 고정으로 부여하는 것
- 사용자가 추가한 필드가 통계에서 소외되는 구조
- 프리셋이 읽기 전용으로만 존재하는 것 (추가/편집/삭제 불가)

### 통계 시스템

**Do:**
- 모든 필드(시스템 + 사용자 추가)를 분석 대상으로 포함
- 성별 비율, 특성 분포, 거주지 분포 등 실질적 인사이트 제공
- 캐릭터별 사건 연계 비율, 관계도 비율, 데이터 양 등 개별 통계 제공
- 작품별로 다른 필드 구성을 인지하고 적응적으로 분석
- 시각화(차트, 그래프)로 직관적인 정보 전달

**Don't:**
- '데이터 입력량'만 보여주는 통계 (입력 완료율, 필드 채움 비율 등)
- 시스템 기본 필드만 분석하고 사용자 커스텀 필드는 무시하는 구조
- 시각화 없이 숫자만 나열하는 통계

### UX / 조작

**Do:**
- 순서 변경은 드래그앤드롭으로 즉시 반영
- 사건은 연표 그래프에 시각적 표시가 있어 존재 파악 가능
- 대략적 배치 → 세부 조정의 이중 경로 제공
- 작품 선택 시 관련 캐릭터 자동 필터링

**Don't:**
- 저장을 위해 버튼 여러 단계를 거쳐야 하는 플로우
- 일일이 연도를 열어봐야 사건이 있는지 알 수 있는 구조
- 숫자 직접 입력만으로 순서를 조정해야 하는 것

### 이미지 / 카드

**Do:**
- 세계관/작품 카드에 이미지 등록 가능 (직접 등록 / 캐릭터 이미지 연동 선택)
- 카드 이미지 크기를 기본 높이(설명 2줄 기준)에 맞춰 적절히 표시

**Don't:**
- 카드 이미지가 지나치게 작게 표시되는 것
- 설명 유무/길이에 따라 카드 높이가 들쭉날쭉하여 레이아웃이 불안정한 것

## 품질 기준

### 방편식 패치 금지

모든 기능은 완결된 형태로 구현되어야 한다. 임시 땜빵, 하드코딩된 예외 처리, 특정 케이스에만 동작하는 코드는 허용되지 않는다. 기능을 구현할 때는 항상 "이 기능이 어떤 경우에도 제대로 동작하는가?"를 기준으로 점검한다.

### 새 기능 체크리스트

새로운 기능을 추가하거나 기존 기능을 수정할 때 아래 항목을 순서대로 검증한다:

1. **확장성** — 사용자가 이 기능을 커스텀/확장할 수 있는가?
2. **실질성** — 이 기능이 실제로 쓸모 있는 정보/기능을 제공하는가?
3. **완결성** — 방편식 패치나 임시 처리가 없는가?
4. **연결성** — 기존 데이터/기능과 유기적으로 연결되는가?
5. **조작성** — 사용자가 최소 단계로 목적을 달성할 수 있는가?

## 통계 시스템 설계 기준

### 통계의 목적

통계 시스템의 목적은 창작 의사결정을 보조하는 분석 도구이다. 단순히 '얼마나 입력했는가'를 보여주는 것이 아니라, '내 캐릭터 군상에 어떤 편향과 패턴이 있는가'를 발견할 수 있어야 한다.

### 필수 분석 범위

- **전체 캐릭터 비율 분석:** 성별, 나이대, 태그 분포 등
- **필드별 분포 분석:** 모든 필드(시스템 + 커스텀)의 값 분포
- **캐릭터별 개별 통계:** 사건 연계 비율, 관계도 비율, 데이터 완성도
- **작품별 비교 분석:** 작품 간 필드 구성 차이, 캐릭터 특성 비교
- **시각화:** 차트, 그래프 등으로 직관적인 정보 전달

### 적응적 통계 원칙

작품별로 필드 구성이 다를 수 있으므로, 통계 시스템은 정적으로 고정된 분석 항목이 아니라 필드 구성에 따라 동적으로 생성되는 분석을 제공해야 한다. 사용자가 새 필드를 추가하면 해당 필드가 통계에 자동으로 편입되고, 분석 방법(비율/분포/추세 등)도 사용자가 선택할 수 있어야 한다.
