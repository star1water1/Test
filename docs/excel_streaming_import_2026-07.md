# 엑셀 가져오기 스트리밍 리더 설계 (PR②)

## 목적 / 배경

현재 가져오기는 `ExcelImporter.kt:353`에서
`xlsxFile.inputStream().use { WorkbookFactory.create(it) }` 로 **파일 전체를 메모리에
DOM(XSSFWorkbook)으로 적재**한다. 이어지는 분석(analyzeAll)·가져오기(importAll)
파이프라인은 이 인메모리 워크북 위에서 `getSheet(name)`, `getRow(i)`, `getCell(idx)`
같은 **랜덤 액세스**로 17개 시트를 처리한다.

캐릭터 수백 명 × 커스텀 필드 수십 열 + 연표·관계 시트가 큰 파일에서는 이 DOM 적재가
`largeHeap`으로도 OOM을 낼 수 있다(PR①에서 largeHeap·용량 상한 완화·OOM 안내로 완충만
해둔 상태). PR②는 근본 원인 — **파일 크기에 비례하는 메모리 사용** — 을 없앤다.

개발 의도 검증(4문항):
- **받쳐주는 확장성**: 대형 파일에서도 메모리가 파일 크기에 비례해 터지지 않아야 진짜 확장이다.
- **변수 제어 / 엑셀 왕복 무결성**: 가져오기는 데이터 무결성의 핵심 경로다. 스트리밍 전환이
  기존 왕복(내보내기→편집→들이기) 정확성을 **한 치도** 바꿔서는 안 된다.

## 제약 (반드시 설계에 반영)

1. **로컬 빌드 불가**: 이 환경은 Android SDK 설치가 프록시(dl.google.com 403)로 막혀 있어
   `assembleDebug`를 로컬에서 돌릴 수 없다. **CI가 유일한 빌드 게이트**이며, CI도 가져오기의
   *의미적* 정확성(무결성)은 잡지 못한다. → 스트리밍 리더는 **순수 JVM 단위 테스트로
   CI에서 실검증 가능한 형태**로 만들어야 한다(인메모리 워크북 생성→스트리밍 read→행 대조).
2. **왕복 무결성 불변**: per-row 파싱 로직(getCellString/getCellCode/parseNumber/parseBoolean 등)은
   **재사용**해야 하며, 스트리밍용으로 별도 파싱을 두어 로직이 갈라지면(drift) 무결성이 깨진다.
3. **POI 5.3.0** (poi + poi-ooxml) 사용 중 — `XSSFReader` + `ReadOnlySharedStringsTable` +
   `XMLReader(SAX)` 이벤트 모델 사용 가능.

## 전략: 단계적 · 이중 경로 · 로직 비분기

### 왜 전면 SAX 재작성이 아닌가
가져오기 서비스는 3,000줄이 넘고 전부 `ss.usermodel`(Sheet/Row/Cell) API에 맞춰져 있다.
이를 통째로 SAX 이벤트 소비로 재작성하면 diff가 거대하고, 로컬 테스트가 불가능한 상태에서
무결성 회귀를 넣기 쉽다(변수 제어 위반 위험). **채택하지 않는다.**

### 채택: 스트리밍 행 추상화 + 이중 경로
1. **행 추상화 `RowData`**: `Map<Int, String>`(열 인덱스→셀 문자열) 기반의 경량 행. 기존 코드가
   실제로 쓰는 셀 접근은 `getCellString(row, idx)`/`getCellCode(...)`/`cellType==NUMERIC` 정도로
   좁다. 이 접근면만 `RowData`로 대응하면 per-row 로직을 그대로 태울 수 있다.
2. **`StreamingSheetReader`**: `XSSFReader`로 시트별 InputStream을 열고 SAX로 **전방향 1회** 순회,
   행마다 `RowData`를 방출(콜백/시퀀스). 공유문자열은 `ReadOnlySharedStringsTable`.
   숫자/날짜 셀은 원시 문자열로 방출하되, 코드가 기대하는 표현과 **동일하게** 정규화
   (기존 getCellString의 숫자·날짜 처리와 1:1 대응 — 이 동치성이 테스트의 핵심).
3. **이중 경로 (안전)**: 기존 DOM 경로를 **기본/폴백으로 유지**하고, 스트리밍은
   (a) 큰 파일(임계값, 예: 파일 크기 or 엔트리 크기) 또는 (b) 설정 토글에서 활성화.
   기존 경로가 검증된 기준으로 남으므로, 스트리밍에 미지의 문제가 있어도 회귀 반경이 제한된다.
   두 경로 모두 **동일한 per-row 로직**을 호출하므로 무결성은 구조적으로 보존된다.

### 시트 처리 호환성
- 가져오기는 의존성 순서(세계관→작품→필드→캐릭터→연표→…)로 **시트별 1회 순회**한다.
  전방향 스트리밍과 정확히 부합.
- 헤더 행(0행) 먼저 읽어 열 인덱스 해소 → 이후 데이터 행 순차 처리: 스트리밍과 부합.
- `lastRowNum`(진행률 총계)은 스트리밍이 끝나기 전엔 모른다 → 값싼 사전 카운트 패스 또는
  근사 진행률로 대체(진행률은 무결성과 무관).
- 다중 캐릭터 시트/세계관별 시트 탐색: `XSSFReader` 시트 반복자가 시트명을 제공 → 부합.

## 단계 (각각 CI 게이트 · 병합 · 재정렬)

- **PR②-a — 스트리밍 리더 기반 (이번)**: `StreamingSheetReader` + `RowData` 신설, 기존 import 경로는
  **건드리지 않음**(순수 추가). **JVM 단위 테스트**로 실검증: 인메모리 XSSFWorkbook에 알려진 값
  (숫자·문자·빈칸·전각·긴 문자열·공유문자열 중복)을 써서 bytes로 만들고, 스트리밍 리더로 읽어
  DOM 리더(getCellString)와 **셀 단위 동치**임을 단언. → CI에서 실제로 검증되는 저위험 토대.
- **PR②-b — 가져오기 경로 배선**: 시트 루프를 `RowData` 소비로 리팩터(로직 불변), 이중 경로 게이트.
  per-sheet 정적 리뷰 + 동치 테스트 확장.
- **PR②-c — 활성화 정책 · 진행률 · 계측**: 임계값/토글, 진행률 총계, 메모리 계측·로깅.

## 위험 / 완화

| 위험 | 완화 |
|---|---|
| 로컬 빌드·기기 테스트 불가 → 무결성 회귀 미검출 | 순수 JVM 동치 테스트(리더 vs DOM), 이중 경로 폴백, per-sheet 리뷰 |
| 숫자·날짜·공유문자열 표현 불일치로 값 왜곡 | 리더가 getCellString과 **동일 정규화**, 테스트가 동치를 강제 |
| 전방향 제약으로 기존 랜덤액세스 패턴 파손 | 시트별 1회 순회로 재구성(현 루프와 동형), 헤더 선독 |
| 진행률 총계 부재 | 사전 카운트 or 근사(무결성 무관) |

## 새 기능 체크리스트
1. 확장성 ✓ (메모리 파일크기 비례 제거) 2. 실질성 ✓ (실제 OOM 원인 제거)
3. 완결성 — 이중 경로는 임시 땜빵이 아니라 **검증 자산 보존**을 위한 설계 선택(로직 비분기로 무결성 유지)
4. 연결성 ✓ (기존 per-row 로직 재사용) 5. 조작성 ✓ (사용자에겐 투명, 임계값 자동)
6. 의도 부합 ✓ (받쳐주는 확장성 + 변수 제어/왕복 무결성 최우선)
