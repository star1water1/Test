## D-1. 가져오기/복원 크기 상한 재설계 [P0-1, P1-1] — 기준 ①②④

**문제:** 백업/내보내기 생성에는 크기 상한이 없는데 가져오기/복원에만 50MB 전체 파일 상한이 있어, 이미지가 많은 사용자의 백업·내보내기 산출물이 스스로 복원 불가가 된다.

**설계:**
1. 상한의 목적(ZIP bomb·자원 고갈 방어)을 "파일 전체 크기"에서 "실제 위험 지점"으로 옮긴다.
2. `importFromLocalFile()`(앱이 생성한 백업의 복원 경로): 파일 전체 상한 **제거**.
3. `importFromUri()`(외부 파일): 전체 상한 50MB → **512MB 상향** (이미지 포함 ZIP 왕복 보장).
4. ZIP 해제 지점에 진짜 방어 추가 (`importFromZip`):
   - `data.xlsx` 엔트리 해제 크기 상한 50MB (POI 메모리 보호) — `util/copyWithLimit` 재사용
   - 이미지 엔트리 개수 상한 2,000개, 이미지 총 해제 용량 상한 1GB
   - 초과 시 구체적 메시지("백업의 데이터 시트가 50MB를 초과합니다" 등)
5. 비-ZIP(xlsx 단독) 경로: 기존 50MB 유지 (POI 파싱 메모리 보호 — 동일 수치).

**대안 검토:** 전체 상한만 상향(1안)은 데이터 시트 50MB 초과 시 POI OOM 위험이 남음 → 기각. 상한 완전 제거(2안)는 zip bomb 방어 상실 → 기각.
**리스크:** 대용량 파일 복사 중 캐시 공간 부족 → 기존 일반 실패 처리로 흡수(메시지는 기존 유지). 스트리밍 복사라 메모리 안전.
**검증 방법:** 코드 경로 추적으로 (a) 백업 생성 크기 상한 없음 ↔ 복원 수용 상한의 정합, (b) zip bomb 시나리오(허위 엔트리 크기 포함) 차단 확인.

## D-2. 기기 종속 암호화 백업 고지 [P0-3] — 기준 ②④

**문제:** .enc 백업이 AndroidKeyStore 기기 종속 키로 암호화되는데 UI에 아무 고지가 없어, 폰 교체 시 백업만 믿던 사용자가 전체 데이터를 잃는다.

**설계:**
1. '백업 내보내기' 실행 전 다이얼로그: "이 백업 파일은 암호화되어 **이 기기에서만** 복원할 수 있습니다. 기기 이전(폰 교체)용으로는 '엑셀로 내보내기'(이미지 포함 ZIP)를 사용하세요." — [계속] / [취소].
2. 복원 시 복호화 실패(GCM 태그 불일치 등) 메시지 구체화: "복원 실패: 이 백업은 다른 기기에서 만들어졌거나 손상되었습니다. 백업은 만든 기기에서만 복원할 수 있습니다."
3. 설정 화면 백업 섹션에 상시 캡션 1줄: "자동/수동 백업은 이 기기 전용입니다. 기기 이전은 엑셀 내보내기를 이용하세요."

**대안 검토:** 패스프레이즈 기반 이식 가능 암호화 백업(PBKDF2+AES-GCM)이 근본 해법이나 신규 기능 규모 — **후속 과제**로 이관. 고지만으로도 '조용한 유실'은 제거됨(변수 제어 원칙의 '알림' 단계 충족).
**리스크:** 없음(문구·다이얼로그 추가).
**검증 방법:** 내보내기·복원·설정 3경로 모두에서 고지가 노출되는지 코드 확인.

## D-3. 엑셀 임포트 무통보 실패 제거 [P0-2, P1-4, P1-5] — 기준 ②④

**문제:** (a) 첫 컬럼 훼손 시 시트 전체 무통보 스킵, (b) 필수 컬럼 부재 시 하드코딩 인덱스 폴백이 이웃 컬럼을 오기록, (c) 숫자 파싱 실패 행이 집계 없이 소멸, (d) 인식 안 된 시트(개명된 캐릭터 시트) 무통보 무시, (e) NUMBER 필드 임포트 타입 검증 없음.

**설계:**
1. **헤더 검증 실패 보고:** `isValidHeader` 실패로 시트를 건너뛸 때 오류 수집: "시트 'X': 첫 번째 컬럼이 '이름'이어야 합니다(현재: 'Y'). 시트를 건너뛰었습니다."
2. **하드코딩 인덱스 폴백 제거:** `cols["사건 설명"] ?: 4` 류의 폴백을 제거하고, 필수 컬럼이 헤더에서 해석되지 않으면 해당 시트를 오류 보고 후 스킵(오기록 방지가 우선). 선택 컬럼은 null 허용 유지. 헤더 별칭 매칭은 기존대로.
3. **숫자 파싱 실패 집계:** `?: continue` 지점(연표/상태변화/관계변화 연도 등)에서 스킵 시 오류 수집: "시트 'X' 행 N: 연도 '값'을 숫자로 해석할 수 없어 행을 건너뛰었습니다."
4. **미인식 시트 알림:** 임포트 종료 시 워크북에서 처리되지 않은 시트 목록을 경고로 보고: "시트 'Z'는 인식되지 않아 무시되었습니다. 세계관 이름과 일치하는지 확인하세요." (사용안내 등 시스템 시트는 제외)
5. **NUMBER 필드 값 검증:** 비숫자 값은 **저장하되 경고 수집**("행 N: 숫자 필드 'F'에 숫자가 아닌 값 '...'이 저장됨 — 통계에서 제외될 수 있음"). 거부가 아닌 수용+알림(의도 ④의 "유연한 수용·교정").

**대안 검토:** 폴백을 유지하고 경고만 추가하는 안은 '이웃 컬럼 오기록'이라는 조용한 데이터 왜곡을 남김 → 기각.
**리스크:** 기존에 폴백 덕에 '우연히' 임포트되던 파일이 스킵될 수 있음 — 오류 메시지가 원인·교정법을 정확히 알려주므로 수용 가능(조용한 오기록보다 낫다).
**검증 방법:** 각 실패 모드별로 오류/경고가 결과 요약에 도달하는 경로 추적. 기존 정상 파일 경로의 무회귀 확인.

## D-4. 내보내기 셀 길이 가드 [P1-6] — 기준 ①④

**문제:** DB 값이 XLSX 셀 한도(32,767자)를 넘으면 POI 예외로 내보내기 전체가 원인 불명 실패. 들이기 10,000자 잘림도 안내 시트에 미고지.

**설계:**
1. 셀 기록 공통 헬퍼에 32,767자 초과 시 잘라 기록 + 잘림 건수·위치(시트/행) 수집.
2. 내보내기 완료 토스트에 잘림 요약 반영: "내보내기 완료 (셀 N개가 엑셀 한도로 잘렸습니다)".
3. 사용안내 시트에 한도 명시: "셀당 최대 32,767자(엑셀 규격), 가져오기 시 필드값은 10,000자까지 저장됩니다."

**대안 검토:** 초과 시 전체 실패 유지는 '받쳐주지 못하는 확장' 그 자체 → 기각.
**리스크:** 잘린 파일을 재들이기하면 잘린 값이 원본을 덮어씀 — 단, 들이기 자체가 10,000자에서 자르므로 기존 동작 대비 악화 없음. 잘림 경고로 인지 가능.
**검증 방법:** 32,767자 초과 값 기록 경로 추적, 정상 값 무영향 확인.

## D-5. 수식 필드의 수식 필드 참조 (중첩 평가) [P1-10] — 기준 ①②

**문제:** CALCULATED 값은 DB에 저장되지 않는데 `resolveField`는 저장값 맵만 보므로 수식이 수식을 참조하면 항상 0. 순환 참조 가드(`resolvingFields`)는 도달 불가한 죽은 코드.

**설계:**
1. `FormulaEvaluator.resolveField(key)`: 해당 key의 FieldDefinition이 CALCULATED 타입이면 **저장값 대신 그 필드의 수식을 재귀 평가**한다. (엑셀 임포트로 유입된 낡은 정적 값보다 항상 신선한 계산이 우선 — P2-10 일부도 함께 완화)
2. 재귀 진입 시 `resolvingFields`에 key 추가, 종료 시 제거 — 순환 참조는 NaN 반환(기존 가드가 비로소 작동).
3. 평가 결과 memoization(인스턴스 단위 캐시)로 동일 필드 반복 평가 방지.

**대안 검토:** CALCULATED 값을 DB에 구체화(materialize)하는 안은 갱신 시점 관리 복잡도가 크고 기존 "저장 안 함" 설계와 충돌 → 기각.
**리스크:** 깊은 중첩 체인의 성능 — memoization + 순환 가드로 상한이 잡힘. TimeStateResolver·통계·엑셀 내보내기 등 모든 호출부가 동일 로직을 공유하므로 일관성 확보, 단 호출부별 fieldDefinitions 전달 여부 확인 필요.
**검증 방법:** A→B→C 중첩, A→B→A 순환, 미존재 키 3케이스의 평가 경로 추적. 기존 단순 수식 무회귀.

## D-6. 수식 저장 시 검증 경고 [P1-12] — 기준 ②③

**문제:** 수식 오타(괄호 누락·미존재 키)가 무검증 저장되어 조용히 NaN/0 처리됨.

**설계:**
1. FieldEditDialog 저장 시 CALCULATED 타입이면: (a) 시험 평가(전 필드 0.0 컨텍스트)로 구문 오류(NaN) 검출, (b) `field('키')` 참조를 추출(기존 `getReferencingCalculatedFields` 정규식 재사용)해 미존재 키 검출.
2. 문제 발견 시 **차단하지 않고** 경고 다이얼로그: "수식에 문제가 있을 수 있습니다: [원인 목록]. 그대로 저장할까요?" — [저장] / [수정하기]. (아직 만들지 않은 필드를 나중에 만드는 작업 순서를 존중 — 자율성 우선)

**리스크:** 시험 평가의 오탐(0 나눗셈 등 값 의존 NaN) — 경고 문구를 "문제가 있을 수 있습니다"로 완곡화하고 저장 선택권을 사용자에게 둠.
**검증 방법:** 정상 수식 무경고, 괄호 누락/미존재 키 경고 발생, 저장 선택 시 정상 저장 확인.

## D-7. GRADE 커스텀 등급 보존 [P1-13] — 기준 ①②

**문제:** 런타임은 임의 등급 라벨(D~SS 등)을 지원하는데, 필드 편집 다이얼로그의 `buildConfig`가 C/B/A/S 4키로 grades를 재작성해 엑셀로 정의한 커스텀 등급을 무통보 파괴.

**설계:**
1. `buildConfig`의 GRADE 처리: 기존 config의 grades 맵을 파싱해 **C/B/A/S 외 키를 보존**하고 UI의 4키 값만 갱신.
2. 커스텀 등급이 존재하는 필드를 열면 다이얼로그에 안내 1줄: "커스텀 등급 {D, SS}는 유지됩니다 (편집은 엑셀 필드정의 시트에서)."

**대안 검토:** 등급 편집 UI를 동적 리스트(추가/삭제)로 개편하는 근본 해법은 **후속 과제** (UI 규모).
**리스크:** 없음에 가까움 — 보존 로직은 순수 병합.
**검증 방법:** 커스텀 등급 필드 저장 왕복 후 config 보존 확인 경로 추적.

## D-8. 삭제 영향 고지 [P1-15, P1-16] — 기준 ②

**문제:** 캐릭터/세계관 삭제 확인이 연쇄 삭제 범위를 알려주지 않음 (undo도 없으므로 사전 고지가 유일한 방어선).

**설계:**
1. **캐릭터 삭제:** 확인 다이얼로그 표시 전 집계 조회 → "관계 N건, 상태변화 이력 M건, 세력 소속 K건, 이미지 L장이 함께 영구 삭제됩니다." (필요 DAO count 쿼리 추가; 0건 항목은 생략)
2. **세계관 삭제:** "작품 N개의 세계관 연결이 해제되고, 필드 정의 M개와 이 세계관 캐릭터들의 필드값 X건·상태변화 이력이 영구 삭제됩니다."
3. 일괄 삭제(batchDelete) 경로도 동일 요약 적용.

**대안 검토:** 휴지통/undo가 근본 해법이나 데이터 모델 신설 규모 → **후속 과제**. 고지는 그 전 단계의 최소 완결.
**리스크:** 다이얼로그 표시 전 suspend 집계 1회 — 기존 lifecycleScope 패턴으로 흡수, 지연 무시 가능.
**검증 방법:** 집계 쿼리 정확성(CASCADE 대상과 일치) 대조.

## D-9. 회전 시 동적 필드 입력 유실 레이스 수정 [P1-2] — 기준 ②

**문제:** 회전 복원용 `pendingFieldValues`가 작품 스피너의 position 0 조기 콜백에서 `fieldDefinitions`가 빈 상태로 소비·소거되어, 이후 실제 폼 구성 시 DB 값으로 덮어씀.

**설계:**
1. 소비 조건 강화: `fieldDefinitions`가 비어 있으면 소비하지 않고 유지(소거하지 않음).
2. 실제 소비는 동적 폼이 구성된 뒤(필드 정의 로드 완료 콜백)에만 수행하고, 그때 null 처리.

**리스크:** 소비 시점이 늦어져도 저장 전 입력이므로 부작용 없음. 새 캐릭터 경로(이미 안전)는 무변경.
**검증 방법:** 기존 캐릭터+회전, 새 캐릭터, 작품 변경 3경로의 소비 흐름 추적.

## D-10. 필드 키 변경 시 참조 마이그레이션 [P1-14] — 기준 ②

**문제:** 필드 key 변경 시 (a) `field('옛키')` 참조 수식, (b) `CharacterStateChange.fieldKey` 문자열 이력이 무통보 파손.

**설계:**
1. 저장 시 key 변경 감지 → 참조 수식 수(기존 스캐너 재사용)와 상태변화 이력 수(fieldKey=옛키, 해당 세계관 캐릭터 한정) 집계.
2. 둘 다 0이면 그대로 저장. 1건 이상이면 다이얼로그: "키를 '{old}'→'{new}'로 바꾸면 수식 N개와 상태변화 이력 M건을 자동으로 함께 갱신합니다. 계속할까요?" — [계속(자동 갱신)] / [취소].
3. 계속 시 단일 트랜잭션: 수식 텍스트 치환(정규식 `field\(\s*['"]?old['"]?\s*\)` 3형태 정확 매칭), `character_state_changes.fieldKey` 일괄 UPDATE.

**리스크:** 치환 오탐 — 정규식을 완전 일치 형태로 한정(부분 문자열 오탐 차단). 상태변화 UPDATE 범위(세계관 경계) 정확성은 적대적 검증 대상.
**검증 방법:** old가 다른 키의 부분 문자열인 케이스, 따옴표 3형태, 세계관 경계 케이스 추적.

### 2차 배치 구현 메모 (2026.07.03)

- **B-1**: `BackupEncryptor`에 이식 가능 형식 "NCP1" 추가 — 헤더 `MAGIC(4B)+iterations(4B)+salt(16B)+chunkSize(4B)`, PBKDF2WithHmacSHA256(200k회)로 AES-256 키 유도, 기존 청크 GCM 구조 재사용. 내보내기 시 형식 선택(이식 가능/기기 전용), 복원 시 매직 자동 감지 → 암호 입력, 오답(AEADBadTagException)은 재입력으로 유도. 기기 키 소실 시에도 이식 백업 복원 경로는 열어둠.
- **B-2**: `EventEditDialogFragment` 신설(구 `EventEditDialogHelper` 삭제). 정적 입력은 뷰 상태 자동 복원, 체크 선택은 `onSaveInstanceState`(LongArray), 데이터 접근은 호스트의 `Host.eventDialogDataProvider()`로 재생성 후 재획득. 간편 추가도 `QuickAddEventDialogFragment`(FragmentResult)로 전환. `maxDayOfMonth`/`isValidDay`는 `util/DateValidation.kt`로 이동.
- **B-6**: `CharacterDraftPrefs`(SharedPreferences JSON) 신설. `onPause`에서 미저장 변경 시 드래프트 기록, 재진입 시(savedInstanceState 없음) 복원/삭제/나중에 제안. 저장 성공·명시적 폐기 시 드래프트 제거. 태그 입력에도 변경 추적 추가(기존 공백 보완). 동적 필드 복원은 기존 `pendingFieldValues` 지연 소비 메커니즘 재사용.
